{{ _('Domain') }}

各ユーザーは自身のドメインを作成できます。例えば、教師は授業のためにドメインを作成し、問題や学生をそのドメインに追加できます。また、ドメインをチーム機能として利用することも可能です。

役割とその権限を適切に設定することで、ドメインを公開または非公開に設定できます。

デフォルトドメインは system です。ユーザーが {{ name }} のドメインに直接アクセスすると、このドメインにアクセスします。Hydro のロゴをクリックすると、どのドメインからでも system ドメインに戻ります。

各ドメインの問題集、ディスカッション、トレーニング、およびコンテストは独立しており、自己管理されています。

{{ _('Limitations') }}

{{ name }} のジャッジシステムは、プロセスの CPU 時間を使用して時間消費を計算します。時間の制限は、問題のテストケースで指定された時間です。

{{ name }} のジャッジシステムは、プロセスの仮想メモリと物理メモリの合計を使用してメモリ空間の消費量を計算します。メモリ空間のデフォルト制限は 256MiB です。問題で特に指定されている場合は、問題のテストケースで指定されたメモリ空間に限定されます。

{{ _('IO') }}

特別な指示がない限り、{{ name }} は標準入出力（コンソール入出力、画面入出力、STD I/O）を使用します。

{{ _('Judge Status') }}

  • Waiting 採点中：採点リクエストがジャッジシステムで処理されるのを待っています
  • Fetched 採点中：採点リクエストがジャッジシステムで取得され、採点開始の準備中です
  • Compiling 採点中：コンパイル中です
  • Judging 採点中：コンパイルに成功し、採点中です
  • Accepted 合格：プログラムの出力が完全に正しいです
  • Wrong Answer 不合格：プログラムの出力が標準の答えと一致しません（行末のスペースやファイル末尾の空行を除く）
  • Time Limit Exceeded 不合格：プログラムの実行時間が問題の制限を超えました
  • Memory Limit Exceeded 不合格：プログラムの実行メモリ空間が問題の制限を超えました
  • Runtime Error 不合格：プログラムの実行時エラー（配列の範囲外アクセス、ゼロ除算、演算オーバーフロー、スタックオーバーフロー、無効なポインタなど）
  • Compile Error 不合格：コンパイルに失敗しました
  • System Error エラー：システムエラー（この問題が発生した場合は、ディスカッションフォーラムで報告してください）
  • Canceled その他：採点がキャンセルされました
  • Unknown Error その他：不明なエラー
  • Ignored その他：無視されました
「成績取り消し」と表示されている場合、管理者がこの記録を手動で取り消し済みであることを意味します。これは、コードが他のユーザーのコードと非常に似ていることが判明するなど、利用規約に違反した可能性があります。

{{ _('Compile Error') }}

以下の状況が考えられます。

  • 1. 提出時にプログラミング言語を間違って選択した
  • 2. Java のメインクラス名が "Main" ではない
  • 3. C/C++ の場合：以下を参照
  • 4. 一般的なコンパイルエラー

C/C++ ユーザーへの特別な注意：

  • 1. GNU C++ では __int64 は long long 型として記述する必要があります
  • 2. main() の戻り値は void ではなく int として定義する必要があります
  • 3. "for (int i = 0...)" のような文では、for 文の後のインデックス変数 i は無効になります
  • 4. itoa は一般的な ANSI 関数ではありません（標準 C/C++ にはこの関数はありません）
  • 5. printf で %lf 形式を使用するのは正しくありません

{{ _('Contest') }}

コンテスト形式によって、提出やランキングのルールが異なります。

OI 形式では、すべての問題は最後の提出に基づいて採点されます。特に、コンパイルエラーを避けるようにしてください。

OI 形式のランキングルール：合計点が高い順に順位が決まります。合計点が同じ場合は同じ順位になります。

ACM/ICPC 形式では、すべての問題は提出後すぐに採点され、合格かどうかが基準となります。

ACM/ICPC 形式のランキングルール：合格した問題数が多い順に順位が決まります。合格した問題数が同じ場合は、問題解決にかかった時間（ペナルティ時間を含む）が少ない方が上位になります。

{{ _('Ledo') }} 形式では、選手は1つの問題に複数回提出でき、リアルタイムの採点結果を取得できます。

{{ _('Ledo') }} 形式では、複数回の提出により選手の得点が減点されます。ランキングにはユーザーの最高得点が表示されます。

{{ _('Ledo') }} 形式では、各問題の最終得点は $s \times \max(0.95^{n}, 0.7)$ です。$s, n$ はそれぞれ今回の得点と今回の提出前の試行回数を表します。

{{ _('Ledo') }} のランキングルール：上記のルールで換算された点数が高い順に順位が決まります。

{{ _('IOI(Strict)') }} 形式では、IOI 形式とは異なり、コンテスト終了までランキングが非公開となります。

{{ _('IOI(Strict)') }} 形式では、各問題のランキング得点は、すべての提出の中でユーザーの各サブタスクの最大得点の合計となります。

時間と空間の制限は問題の説明に従ってください。デフォルトの制限は{{ _('Limitations') }}を参照してください。

{{ _('Accepted Ratio') }}

正答率は悪影響が大きいため、{{ name }} では正答率を提供も承認もしていません。

{{ _('RP Algorithm') }}

RP は、問題、アクティビティ、コンテストなどから獲得できます。

{{ _('Difficulty Algorithm') }}

{{ name }} の問題の難易度は、提出数、正答率、および各提出の提出時間と採点結果に基づいて、アルゴリズムによって計算されます。

したがって、以下の点に注意してください。

  • 0. 一般的に、難易度の数値が大きいほど、問題は難しくなります。
  • 1. 新しい問題の難易度は不正確な場合があります。多くの提出が行われた後に、難易度はより正確になります。
  • 2. 採点のための提出が早いユーザーのコードの採点結果は、問題の難易度に大きな影響を与えます。
  • 3. 問題の難易度はアルゴリズムによって計算されるため、不正確な結果になる可能性があります。

{{ _('Markdown') }}

Hydro の Markdown 領域は CommonMark 構文を使用し、以下の機能が拡張されています。

  • Katex 構文に基づいた数式に対応。
  • `@[](/user/uid)` を使用してユーザーに言及。
  • `==text==` を使用してテキストをハイライト。
  • `![alt](url =100x100)` を使用して埋め込み画像のサイズを設定。
  • `@[video](https://.../a.mp4)` を使用して動画ファイルを埋め込み。
    • {#
  • `@[bilibili](BVxxx)` を使用して外部動画を埋め込み。
    • #}
  • `@[pdf](https://.../a.pdf)` を使用して PDF ファイルを埋め込み。
  • 問題/コンテスト/課題/トレーニングでは、`file://ファイル名` のリンク形式を使用して、対応する添付ファイルを参照できます。（推奨）
  • 一部の HTML タグの使用をサポート。
